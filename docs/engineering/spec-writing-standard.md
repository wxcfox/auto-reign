# Spec 编写规范

本文规定 Auto Reign 中较大行为变更的设计说明应回答什么问题，以及设计、实施计划和长期文档之间如何分工。目标是让开发者和 Agent 在同一组可验证契约下协作，而不是依赖口头上下文或未经验证地沿用外部方案。

## 适用范围

出现以下任一情况时，修改应用代码前必须先形成 Spec：

- 改变 Agent、Task、Subtask、Context、Knowledge、Document 或用户的生命周期；
- 改变 MySQL、Redis、ObjectStore、Elasticsearch 或 Qdrant 的权威边界；
- 新增或修改跨数据库、后端、API、Socket.IO 与前端的契约；
- 引入状态机、后台任务、并发控制、重试、恢复或数据迁移；
- 改变权限、租户隔离、Prompt、工具能力或主界面主要流程；
- 需要分阶段交付，或者一次变更无法由单个聚焦测试完整说明。

局部重命名、纯样式调整和不改变外部行为的内部重构可以不单独写 Spec，但仍须遵守现有架构与测试规范。

## 文档职责

不同文档各自只承担一种职责：

| 文档 | 职责 |
| --- | --- |
| `README.md` | 当前可运行能力、启动方式、配置和开发检查 |
| `docs/workbench-architecture.md` | 平台长期架构、不变量与权威边界 |
| `docs/*.md` | Knowledge、生产部署等当前有效的专题流程 |
| `docs/superpowers/specs/*.md` | 已批准但尚需独立表达的复杂行为契约或设计记录 |
| `docs/superpowers/plans/*.md` | 从已批准 Spec 拆出的临时实施任务 |

实施计划不是第二份架构文档。功能完成后，应把仍然有效的事实沉淀到权威长期文档，并删除已经完成的一次性计划。

## 证据优先

Spec 必须先描述当前实现，再提出目标。不得只依据记忆、名称相似或参考项目文档推断行为。

每份 Spec 至少记录：

- Auto Reign 当前相关代码、数据库模型、API、前端类型、测试和长期文档的位置；
- 当前正常流程、失败路径和已知限制；
- 如采用行业标准、成熟实现或仓库既有模式，指出对应规范、源码或测试，以及准备吸收的行为契约；
- Auto Reign 的方案取舍，并说明保留差异或改变现状的理由；
- 参考实现中仅为历史兼容、旧基础设施或当前项目不使用的字段与分支。

参考材料只用于验证行为和工程取舍，不得复制其他产品的标识、用户文案、无关字段、兼容层或历史包袱。采用或偏离参考行为时，Spec 都必须明确列出证据、原因和影响；不能在实施阶段悄悄改变已批准契约。

## 规范用语

使用可以验收的陈述：

- “唯一权威来源是……”
- “只有……可以修改该状态。”
- “发生……时必须……”
- “最终写入前必须再次校验……”
- “禁止……”
- “以下场景不在本次范围内……”

避免使用“尽量”“适当”“一般来说”“高内聚低耦合”之类无法判定是否完成的表述。涉及状态、身份、顺序、超时、数量或大小时，应给出明确字段、条件或配置来源。

## 必备章节

允许按需求规模合并相邻章节，但不能遗漏其要回答的问题。

### 1. 背景与当前实现证据

- 用户问题或工程问题是什么；
- 当前实现实际如何工作；
- 相关文件、测试和数据结构在哪里；
- 当前行为为什么不足，哪些现状必须保留。

### 2. 实现依据与方案取舍

- Auto Reign 当前源码和测试表现出的行为是什么；
- 适用的标准、成熟实现或仓库既有模式提供了什么证据；
- 哪些行为契约适合当前产品，哪些能力与当前范围无关；
- 本设计选择、裁剪或增强了什么，为什么。

没有额外参考依据时应明确写“仅依据当前实现”或“不适用”，避免读者误以为已经完成外部验证。

### 3. 目标

用可测试的结果描述完成状态。目标应覆盖用户可见行为、持久化结果和故障后的收敛结果。

### 4. 非目标

明确本次不解决的场景、暂不支持的部署规模、不会新增的兼容层以及不会变更的数据域。非目标不能与必需的安全或数据完整性要求冲突。

### 5. 术语与领域不变量

- 定义代码、数据库、API 和 UI 使用的统一术语；
- 列出身份主键、owner、scope、顺序和生命周期不变量；
- 禁止使用的旧术语或歧义映射；
- 哪些不变量必须由数据库约束、后端校验或类型系统保证。

### 6. 数据权威与持久化边界

对每类数据说明：

- 唯一权威源；
- 可重建投影或临时缓存；
- 创建、更新和删除主体；
- TTL、备份与恢复语义；
- 是否允许丢失；
- 禁止的双读、双写或静默降级。

### 7. 状态机

列出所有状态和允许的转换。每条转换必须说明触发者、前置条件、原子写入、终止状态和异常收敛。派生状态应给出计算规则，不能同时作为另一份可漂移的持久状态。

### 8. 跨层契约

将数据库列、Pydantic schema、HTTP/Socket.IO payload、TypeScript 类型和 UI 展示逐项对齐。至少说明：

- 字段名称、类型、必填性和默认值；
- 事件与 ACK 的顺序；
- 标识符、游标、offset 和幂等键；
- 后端错误码与前端状态；
- 版本演进方式。

### 9. 正常流程

按时间顺序描述从用户动作到最终持久化和 UI 收敛的完整路径。对多存储流程标明事务边界，以及事务外副作用发生的位置。

### 10. 失败、重试、取消、重连与恢复

至少覆盖：

- 请求在持久化前失败；
- 持久化后、执行前失败；
- 执行中连接断开或进程退出；
- 外部 Provider、Redis、Retriever 或 ObjectStore 失败；
- 用户取消和显式重试；
- 页面重连、重复事件和过期缓存；
- 启动恢复如何处理遗留运行态。

每种失败都要说明是否重放、是否创建新记录、旧结果如何处理，以及用户最终看到什么。

### 11. 并发、幂等与 generation

- 同一资源允许哪些并发操作；
- 使用数据库约束、短事务、条件更新还是业务 generation；
- 重复请求和乱序完成如何去重；
- 异步任务最终写入前要重新校验哪些身份、状态和 generation；
- 哪些副作用不能自动重放。

锁只是实现手段，不能代替最终写入校验。没有并发问题时也应明确写“不需要额外锁”及其依据。

### 12. 权限与租户隔离

用矩阵列出角色、资源 scope 和动作。身份必须从认证上下文派生，不能相信客户端提交的 `user_id`。Task room、ObjectStore Key、Retriever filter 和数据库查询应说明如何使用相同的 owner 身份。

### 13. 迁移、重置与兼容结论

- 是否修改 schema；
- 是否要求兼容现有数据；
- 空库和已有库分别如何升级；
- 是否需要显式数据重置；
- 回滚依赖代码回退、备份恢复还是反向迁移；
- 哪些操作可能破坏数据。

Auto Reign 不允许普通启动或 Alembic 静默删除用户数据。破坏性重置只能通过明确命令并由操作者显式确认。

### 14. 可观测性

定义结构化日志事件、必要字段、错误码和审计边界。日志应能关联用户、Task、Subtask、Document generation 或请求，但不得记录密码、Token、API Key、附件正文或其他敏感内容。

### 15. 测试矩阵

逐项映射正常行为、边界条件、失败路径、权限隔离、并发竞争和恢复。说明哪些使用单元或组件测试，哪些必须使用真实 MySQL、Redis、Socket.IO、Elasticsearch 或 Qdrant。测试要求见[测试规范](testing-standard.md)。

### 16. 分阶段实施计划

每个阶段必须：

- 对应明确文件与测试；
- 能独立运行和审查；
- 在阶段边界保持应用可运行；
- 列出迁移影响和验证命令；
- 指明替代完成后删除的旧实现。

### 17. 验收条件

验收项必须可由测试、命令、数据库检查或 UI 操作直接验证。不得用“代码完成”“体验良好”代替结果。

### 18. 旧代码与临时产物清理

列出完成后要删除的旧模型、API、类型、事件、兼容分支、测试夹具和实施计划。没有需要保留的兼容层时，禁止为了“保险”继续双写或双读。

## 评审与变更控制

Spec 评审按以下顺序进行：

1. 核对 Auto Reign 当前实现证据；
2. 核对适用的标准、测试或成熟实现证据是否被准确引用；
3. 确认领域不变量和数据权威；
4. 确认失败、并发、权限与迁移边界；
5. 确认测试矩阵能够证明验收条件；
6. 批准后再拆分实施计划。

实施中发现 Spec 错误或参考实现变化时，先更新并重新确认相关契约，再继续修改代码。实现不得成为未记录的新事实。行为完成后，同一变更必须更新对应长期文档。
